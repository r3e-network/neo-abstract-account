\title{Neo-AA: Cross-Ecosystem Account Abstraction with Heterogeneous Signature Verification}
\author{R3E network\\\small For Neo Ecosystem}
\maketitle

\begin{abstract}
We present \sys, a Neo N3 account-abstraction architecture that unifies native Neo witness authorization with EVM EIP-712 typed signatures inside one shared master contract. The central design challenge is heterogeneity: Neo N3 authorizes transactions through secp256r1-backed witnesses, whereas Ethereum-style workflows authorize through secp256k1 signature recovery over typed data. \sys bridges these models without deploying one wallet contract per user. Instead, it binds each logical account identifier to a deterministic abstract-account address that becomes the user-facing identity, while the logical \code{accountId} remains an internal coordination key for storage, indexing, and verification.

The system contributes four main elements. First, it introduces a mixed authorization protocol that counts Neo witnesses and recovered EVM signer addresses toward one threshold policy. Second, it provides deterministic address generation through a verify-proxy script that routes verification to a shared contract, eliminating per-account deployment costs. Third, it supports human-readable discovery through \code{.matrix} domain resolution and address-based account lookup. Fourth, it exposes a pluggable recovery architecture including Argent-style guardian sets, Safe-style timelocked recovery, Loopring-style guardian-hash recovery, and an oracle-gated dome inactivity flow. We describe the full architecture, state model, workflow, and security properties, and we align the specification with the current implementation and testnet validation pipeline.
\end{abstract}

\section{Introduction}

Account abstraction decouples user identity from cryptographic keys, enabling flexible authorization policies. While Ethereum's ERC-4337 and similar proposals provide single-chain solutions, cross-ecosystem authorization remains unsolved. Neo N3's witness-based transaction model and Ethereum's signature recovery present fundamentally different authorization primitives that cannot be unified through simple bridging.

\subsection{Motivation}

Traditional blockchain accounts tie identity directly to one signing key or to one per-chain smart-wallet deployment. This coupling complicates recovery, fragments user presence across ecosystems, and makes multi-wallet coordination operationally expensive. Neo N3 and EVM ecosystems exacerbate the problem because they expose different authorization primitives, different signing curves, and different transaction assembly workflows.

\subsection{The Neo N3 Challenge}

Neo N3 transactions use a witness array where each witness contains an invocation script and a verification script. Authorization occurs when the verification script (typically checking ECDSA signatures on secp256r1) validates the invocation script against the transaction hash. This model differs fundamentally from Ethereum's approach where signatures are recovered to addresses that are then checked against authorization lists.

\begin{warningbox}[Naive Approaches vs. Neo-AA]
A naive cross-chain account abstraction typically requires:
\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item Deploying separate smart contracts on each chain.
\item Synchronizing authorization state (e.g., via slow cross-chain messaging) across heterogeneous runtimes.
\item Maintaining chain-specific signing flows with no cohesive, multi-chain threshold semantics.
\end{itemize}
These overheads are precisely what \sys\ is designed to eliminate.
\end{warningbox}

\subsection{Challenges}

\begin{warningbox}[Core Architectural Challenges]
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysorange!80!black}]
\item[C1: Heterogeneous Signatures]
Neo N3 uses witness-based authorization with secp256r1, while EVM wallets use EIP-712 typed data and secp256k1 recovery. A unified system must count both schemes without weakening either trust model.

\item[C2: Deterministic Addressing]
Per-account deployment is expensive and operationally noisy. A practical design needs stable account addresses without forcing one contract deployment per user.

\item[C3: Address-First Usability]
Users typically remember an account address or human-readable domain, not a logical identifier. The account model therefore needs a clean distinction between public account identity and internal storage keys.

\item[C4: Recovery and Extensibility]
Production accounts require recoverability, role separation, and policy mutation without hard-coding one recovery scheme into the core contract.
\end{description}
\end{warningbox}

\subsection{Our Solution: Mixed Authorization Protocol}

\sys introduces a \textit{mixed authorization protocol} in which a single Neo N3 master contract accepts both native Neo witnesses and EVM EIP-712 typed signatures and counts them toward one threshold. The key architectural observation is that both signer types can ultimately be normalized into addresses: Neo witnesses are checked with \code{Runtime.CheckWitness}, while EVM signatures are verified on-chain, mapped to recovered Ethereum-style addresses, and evaluated against the same stored role sets.

\begin{conceptbox}[The Neo-AA Approach]
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysblue!80!black}]
\item[Deterministic Addressing]
Instead of deploying one contract per account, \sys derives a deterministic verify-proxy script that forwards verification to the master contract. Internally, each account is keyed by a logical \code{accountId}; externally, users interact with the deterministic abstract-account address produced from that proxy script.

\item[Address-First Routing]
The common execution path exposes both logical-id entrypoints and address-based convenience wrappers. User-facing interfaces rely exclusively on the deterministic AA address---or a bound \code{.matrix} domain---rather than an opaque logical identifier.

\item[EIP-712 Integration]
EVM signers authorize an operation by signing typed data bound to the AA address, target contract, method hash, argument hash, nonce, and deadline. The master contract reconstructs the digest on-chain, recovers signer addresses, and feeds them into the same permission engine used for Neo execution.
\end{description}
\end{conceptbox}

\subsection{Contributions}

This paper makes the following contributions:

\begin{featurebox}[Key Contributions]
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysgreen!80!black}]
\item[Mixed Authorization Protocol]
A unified threshold mechanism that counts Neo witnesses and EVM EIP-712 signatures toward one authorization threshold, enabling cross-ecosystem account abstraction without bridge contracts or synchronized replica state.

\item[Deterministic Address Generation]
A zero-deployment account model where verify scripts are deterministically generated and route to a shared master contract, eliminating per-account deployment costs while maintaining address stability.

\item[Unified Threshold Algorithm]
The \code{CheckMixedSignatures} algorithm seamlessly integrates Neo's \code{Runtime.CheckWitness} with EVM signature recovery, supporting pure Neo, pure EVM, and mixed authorization scenarios through a single verification path.

\item[Formal Security Analysis]
Rigorous security definitions and proofs demonstrating threshold security, replay protection, and authorization correctness under the standard cryptographic assumptions for both signature schemes.
\end{description}
\end{featurebox}
\begin{conceptbox}[EVM Integration and Address-First Architecture]
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysblue!80!black}]
\item[How EVM Participation Works]
EVM wallets join \sys\ through EIP-712 typed-data signatures. When an EVM signer authorizes an operation, the signed message commits to the AA account address, target contract, method hash, argument hash, nonce, and deadline. The signature is carried into a Neo transaction, verified on-chain with secp256k1/Keccak256, and converted into a recovered Ethereum address that is evaluated against the stored admin or manager sets.

\item[Address-First Account Management]
The contract still stores a logical \code{accountId}, but the dominant external interface is address-based. The common creation path is \code{createAccountWithAddress}, which binds the logical record directly to its deterministic proxy address. The contract exposes bidirectional lookup methods like \code{getAccountIdByAddress}.

\item[Role Discovery and Indexing]
Instead of forcing users to remember internal identifiers, the master contract maintains address-centric indexes. The contract natively exposes role-specific queries like \code{getAccountAddressesByAdmin}.
\end{description}
\end{conceptbox}

\subsection{.matrix Domain Integration}

\sys integrates with the Matrix naming system as a discovery layer, not as an authorization root. When a user enters \code{alice.matrix}, the frontend resolves the domain owner from the Matrix contract, queries the AA contract for bound addresses associated with that owner, and presents the discovered AA accounts for selection. A typical resolution flow is:

\begin{conceptbox}[Matrix Domain Lookup Flow]
\begin{enumerate}[leftmargin=*,label=\textbf{\arabic*.}]
\item Resolve the \code{.matrix} domain to its owner address through the Matrix contract.
\item Query \code{getAccountAddressesByAdmin} and \code{getAccountAddressesByManager} on the AA contract using the resolved owner address.
\item Merge, de-duplicate, and present the resulting bound AA addresses.
\item Load the selected AA account and continue all subsequent execution through the account address.
\end{enumerate}
\end{conceptbox}

This separation is intentional: domains improve usability and discovery, while the authoritative authorization state remains on-chain in the AA contract's role and policy storage.

\subsection{Recovery Verifier Architecture}

Neo-AA supports pluggable recovery verifiers through a standardized interface. Each account can optionally set a recovery verifier contract that implements:

\begin{codebox}[language=java]
interface IRecoveryVerifier {
  bool verifyRecovery(
    byte[] accountId,
    byte[] operation,
    byte[] proof
  );
}
\end{codebox}

\begin{conceptbox}[Argent-Style Guardian Recovery]
The Argent recovery model implements a social recovery mechanism where trusted guardians can collectively authorize account recovery operations.

\textbf{On-Chain State Maintained}
\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item \textbf{Guardian Set}: Serialized array of guardian addresses, stored as \code{byte[]} to support both Neo and EVM addresses.
\item \textbf{Guardian Threshold}: Minimum number of guardian signatures required ($m$ out of $n$ guardians).
\item \textbf{Pending Operations}: Map of recovery operation hashes to pending recovery requests.
\item \textbf{Timelock Period}: Configurable delay (default: 24 hours) between initiation and execution.
\end{itemize}

\textbf{Recovery Operation Lifecycle}
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysblue!80!black}]
\item[Phase 1: Initiation]
A guardian initiates recovery via the entrypoint by supplying the \code{accountId}, replacement administrators, and the proposed threshold. The mechanism computes a hash $h = H(\text{accountId} \| \text{newAdmins} \| \text{newThreshold} \| \text{nonce})$, stores the pending operation, and emits a \code{RecoveryInitiated} event.

\item[Phase 2: Guardian Approval]
Additional guardians approve the pending operation by submitting the operation hash alongside their approval signatures. The verifier validates the signature against the hash and increments the approval count.

\item[Phase 3: Execution]
Once the threshold is met and the timelock expires ($\text{Runtime.Time} - \text{initiationTime} \ge T_{\text{lock}}$), anyone can call \code{executeRecovery(operationHash)}. The verifier confirms the conditions and triggers the master contract update.

\item[Cancellation]
The current account owner can reset and cancel any pending recovery operation at any time by successfully performing a standard authorized transaction.
\end{description}
\end{conceptbox}



\begin{conceptbox}[Safe-Style Threshold Timelock]
Extends the core Argent model with enhanced timelock flexibility and modular execution.

\textbf{Key Design Elements}
\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item \textbf{Configurable Timelock}: Each account can set custom timelock periods (1 hour to 30 days).
\item \textbf{Execution Window}: After the timelock $T_{\text{lock}}$ expires, the operation remains valid strictly for a time window $T_{\text{window}}$ (default: 7 days).
\item \textbf{Multi-Step Recovery}: Supports partial admin replacement for a gradual transition rather than complete takeover.
\end{itemize}

The Safe verifier ensures execution fails if $\text{Runtime.Time} > \text{initiationTime} + T_{\text{lock}} + T_{\text{window}}$.
\end{conceptbox}

\begin{conceptbox}[Loopring-Style Guardian Hash]
Prioritizes privacy by hiding guardian identities until the recovery event, storing only a commitment hash on-chain: $H_g = H(\text{guardian}_1 \| \ldots \| \text{guardian}_n \| \text{threshold})$. The verifier also maintains a \code{Recovery Nonce} to prevent replay attacks across multiple recovery attempts.

\textbf{Recovery with Hash Revelation}
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysblue!80!black}]
\item[Phase 1: Initiation]
The initiator calls \code{initiateRecovery(accountId, guardianSet, threshold, signatures)}. The verifier computes $H_{\text{revealed}} = H(\text{guardianSet} \| \text{threshold})$ and confirms $H_{\text{revealed}} = H_g$. It then verifies signatures and stores the pending operation.
\item[Phase 2: Execution]
After the timelock, \code{executeRecovery} verifies the revealed guardian set matches the stored hash and executes the recovery operation.
\end{description}
This design guarantees that guardian addresses cannot be enumerated or analyzed before an actual compromise event forces a recovery.
\end{conceptbox}

\subsection{Dome Inactivity Recovery Protocol}

The dome mechanism provides last-resort recovery for accounts with prolonged inactivity.

\begin{featurebox}[Dome Inactivity Flow]
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysgreen!80!black}]
\item[Inactivity Detection]
The system tracks the last activity timestamp for each account: $\text{lastActivity}[\text{accountId}] = \text{Runtime.Time}$.
An account is flagged as inactive if $\text{Runtime.Time} - \text{lastActivity} > T_{\text{inactivity}}$, where $T_{\text{inactivity}}$ is highly configurable (default: 180 days).

\item[Oracle-Gated Initiation]
A designated recovery address can initiate dome recovery, but execution requires external oracle approval:

\begin{codebox}[language=Java]
function initiateDomeRecovery(accountId) {
  require(msg.sender == domeRecoveryAddr[accountId]);
  require(isInactive(accountId));

  requestId = hash(accountId, Runtime.Time);
  pendingRecovery[requestId] = {
    accountId: accountId, initiator: msg.sender,
    timestamp: Runtime.Time, status: "pending_oracle"
  };

  emit DomeRecoveryInitiated(requestId, accountId);
}
\end{codebox}

\item[Oracle Validation \& Recovery]
An authorized oracle verifies genuine inactivity off-chain and signs an approval. After oracle approval, a final timelock period (default: 7 days) must elapse. During this window, the original owner can unilaterally cancel the recovery by performing any authorized transaction.
\end{description}
\end{featurebox}

\section{Account Model and Lifecycle}

\subsection{Account Creation and Structure}

An abstract account in \sys is a shared-contract wallet instance with two identifiers: a logical \code{accountId} and a deterministic AA address. The \code{accountId} is the internal storage key and verification parameter; the deterministic AA address is the public, user-facing identity. This separation is central to the implementation and to the user experience: users fund, discover, and invoke the AA through its address, while the contract keeps the logical identifier for compact indexing, address derivation, and compatibility with internal verification routines.

\begin{conceptbox}[Identity Model]
\begin{description}[leftmargin=*,style=sameline,font=\bfseries\color{sysblue!80!black}]
\item[Logical identity:] \code{accountId} is created once, validated, and used as the canonical key for durable storage inside the master contract.
\item[Operational identity:] The deterministic AA address is the signer identity that wallets, relayers, and downstream contracts query and interact with.
\item[Binding layer:] The master contract maintains authoritative bidirectional \code{accountId} $\leftrightarrow$ \code{accountAddress} mappings.
\item[User-facing rule:] After creation, all operational pathways strongly prefer the AA address; \code{accountId} is explicitly retained for initialization, storage indexing, and base verifier compatibility.
\end{description}
\end{conceptbox}

\subsection{Deterministic Address Generation}

Each abstract account has a deterministic Neo address derived from its \code{accountId} and the master contract hash. That deterministic address is the account's public identity, signer handle, and discovery key.

\textbf{Verify-Proxy Construction.}

The exact bytecode is produced by the contract's \code{BuildVerifyProxyScript(accountId)} helper. At a logical level, the proxy script performs the following steps:

\begin{codebox}[language=Java]
push accountId
push "verify"
push masterContractHash
syscall System.Contract.Call
\end{codebox}

Conceptually, the proxy script:
\begin{enumerate}
\item Carries the logical \code{accountId} into the verification call.
\item Routes verification to the shared AA master contract rather than to a per-account contract instance.
\item Causes the script hash of the proxy to become the externally visible AA address.
\item Allows downstream contracts to rely on \code{Runtime.CheckWitness(accountAddress)} once an authorized AA execution path has opened the proper verification context.
\end{enumerate}

\begin{conceptbox}[Deterministic Proxy Addressing]
The Neo address is computed as:
$$\text{address} = \text{Base58Check}(\text{version} \| \text{Hash160}(\text{verifyScript}))$$
where \code{version = 0x35} for Neo N3. The script hash is derived from the proxy bytecode and reversed into Neo's \code{UInt160} representation before Base58Check encoding.

\textbf{Key Properties}
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysblue!80!black}]
\item[Deterministic Construction] The same \code{accountId} always yields the exact same verify-proxy address.
\item[Zero Deployment Cost] No per-account contract deployment transaction is required.
\item[Address-First Usability] The account can be funded, discovered, and referenced by address immediately after derivation.
\item[Binding Integrity] The master contract refuses to bind an \code{accountId} to any address other than its mathematically derived proxy address.
\end{description}
\end{conceptbox}

\subsection{Transaction Structure and Verification Entry}

\begin{warningbox}[Transaction Integrity Rule]
\textbf{All transactions using an abstract account MUST include the deterministic verify address as a signer.} This strictly triggers the verification entry point in the master contract.
\end{warningbox}

\begin{featurebox}[Standard Neo-Only Transaction Structure]
\begin{enumerate}[leftmargin=*,topsep=2pt,itemsep=1pt,label=\textbf{\arabic*.}]
\item \textbf{Script}: Invokes the address-based wrapper \code{executeByAddress(accountAddress, targetContract, method, args)}.
\item \textbf{Signers}: Array containing the deterministic verify address (script hash).
\item \textbf{Witnesses}: Array of witness objects containing an empty Invocation Script (or signature data) and the deterministic Verification Script.
\end{enumerate}

\textbf{Verification Entry Flow}
\begin{description}[leftmargin=*,style=sameline,font=\color{sysgreen!80!black}]
\item[1. VM Check] Node executes the verification proxy script, calling \code{masterContract.verify(accountId)}.
\item[2. Guard] The master contract ensures the calling context is a valid self-invocation targeting an approved wrapper.
\item[3. Application Phase] If verified, the transaction script executes the wrapper method.
\item[4. Authorization] The wrapper performs strict authorization checks (admin/manager thresholds) against the on-chain policy.
\item[5. Dispatch] If authorized, the wrapper safely invokes the target contract.
\end{description}
\end{featurebox}

This two-phase design guarantees that the deterministic verify address cannot be misused for unauthorized external calls, and all operations pass through the master contract's centralized policy enforcement layer.

\subsection{Mixed Authorization Transaction Anatomy}

Mixed authorization transactions combine Neo witnesses with EVM EIP-712 signatures. The transaction structure extends the standard Neo transaction with embedded EVM signature data.

\begin{featurebox}[Mixed Transaction Structure]
\begin{enumerate}[leftmargin=*,topsep=2pt,itemsep=1pt,label=\textbf{\arabic*.}]
\item \textbf{Script}: Invokes the address-based meta-transaction wrapper \code{executeMetaTxByAddress(...)}.
\item \textbf{Signers}: Deterministic verify address (same as Neo-only transactions).
\item \textbf{Attributes}: Custom transaction attribute containing EVM signature ($r, s, v$), Nonce, and Deadline.
\item \textbf{Witnesses}: Neo witnesses (may be empty if only EVM authorization is used).
\end{enumerate}
\end{featurebox}

\begin{conceptbox}[EVM Signature Verification Flow]
The master contract's \code{executeMetaTxByAddress} method evaluates EVM payloads via the following sequence:
\begin{description}[leftmargin=*,style=sameline,font=\color{sysblue!80!black}]
\item[1. Reconstruct] Builds the EIP-712 typed data hash from the provided parameters.
\item[2. Recover] Extracts EVM public key: $\text{pubkey} = \text{ecrecover}(\text{hash}, r, s, v)$.
\item[3. Derive] Computes the Ethereum address: $\text{addr}_{\text{EVM}} = \text{Keccak256}(\text{pubkey})[12:32]$.
\item[4. Validate Anti-Replay] Checks nonce matches $n_{\text{stored}}$ and time $\le$ deadline.
\item[5. Map to Roles] Counts recognized EVM signatures alongside any Neo witnesses toward the required threshold.
\end{description}
\end{conceptbox}

\begin{conceptbox}[Neo vs. Mixed Transaction Verification]
While pure Neo transactions authenticate during the memory pool phase, mixed transactions introduce EVM signatures via transaction attributes that must be verified during execution.
\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item \textbf{Payload Location}: EVM signatures are explicitly embedded as parameters within the execution script, not inside the native Neo witness array.
\item \textbf{Timing}: EVM signatures are verified securely on-chain during the application phase, preventing manipulation of the abstract account state.
\item \textbf{Replay Protection}: EVM payloads utilize deliberate, account-scoped nonces embedded in the EIP-712 hash, whereas Neo witnesses are intrinsically bound point-in-time to the transaction hash.
\item \textbf{Unified Policy}: Both paths route into one ultimate counting process against the master policy threshold.
\end{itemize}
\end{conceptbox}

\section{Mixed Authorization Protocol}

\subsection{Protocol Overview}

\begin{featurebox}[Mixed Authorization Protocol Phases]
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysgreen!80!black}]
\item[Phase 1: EVM Signature Generation]
(Off-chain). The EVM signer constructs EIP-712 typed data and signs with their Ethereum wallet. The signature is a 65-byte value $(r, s, v)$ where $r, s \in \mathbb{F}_p$ are ECDSA signature components and $v \in \{27, 28\}$ is the recovery identifier.

\item[Phase 2: Transaction Construction]
A Neo transaction is built with:
\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item \textbf{Script}: \code{CALLT} to master contract wrapper method
\item \textbf{Signers}: Deterministic verify address for the account
\item \textbf{Attributes}: EVM signature embedded as custom transaction attribute
\item \textbf{Witnesses}: Placeholder for Neo signers
\end{itemize}

\item[Phase 3: Neo Witness Collection]
Neo signers add witnesses to the transaction. Each witness contains an invocation script (typically empty or containing the signature) and a verification script (the deterministic proxy script).

\item[Phase 4: On-Chain Verification]
The master contract validates the deterministic proxy witness, counts Neo witnesses, recovers the Ethereum addresses from the EVM signatures, and checks if the combined signature payload hits the unified operational threshold.
\end{description}
\end{featurebox}



\subsection{Core Verification Methods}

The Abstract Account contract implements three fundamental verification methods that form the foundation of the mixed authorization protocol. These methods handle different verification scenarios and execution phases.

\subsubsection{Verify Method: Two-Phase Verification Model}

The \code{Verify(accountId)} method is the unified verification entrypoint called by the deterministic proxy script. It implements a two-phase verification model that distinguishes between transaction validation and application execution.

\textbf{Method Signature:}
\begin{codebox}[language=Java]
public static bool Verify(ByteString accountId)
Location: AbstractAccount.AccountLifecycle.cs:66-113
\end{codebox}

\begin{conceptbox}[Two-Phase Verification Architecture]
The Neo VM verification system operates in two distinct phases, mapping naturally to our mixed-signature routing:

\textbf{Phase 1: Verification Trigger (Transaction Validation Phase)}
\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item \textbf{Timing}: Occurs when a transaction enters the memory pool, before application execution.
\item \textbf{Limitation}: EVM signatures cannot be verified here because EIP-712 signature data resides in transaction parameters, which are inaccessible during the pure Verification phase.
\item \textbf{Behavior}: The \code{Verify} method checks if the transaction shape matches the standard AA wrapper call, then exclusively counts Neo \code{CheckWitness} signatures.
\end{itemize}

\textbf{Phase 2: Application Trigger (Execution Phase)}
\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item \textbf{Timing}: Occurs during the execution phase when the target contract explicitly calls \code{Runtime.CheckWitness(accountAddress)}.
\item \textbf{Behavior}: \code{VerifyContext} ensures only the currently pre-authorized target contract can pass, unlocking EVM typed verification logic.
\end{itemize}
\end{conceptbox}




\subsubsection{CheckNativeSignatures: Pure Neo Signature Verification}

The \code{CheckNativeSignatures} method handles pure Neo signature verification using \code{Runtime.CheckWitness()}. It is used in the Verification phase and in pure Neo execution paths.

\textbf{Method Signature:}
\begin{codebox}[language=Java]
private static bool CheckNativeSignatures(
    List<UInt160> roles, 
    int threshold
)
Location: AbstractAccount.ExecutionAndPermissions.cs:252-261
\end{codebox}

\begin{featurebox}[Pure Neo native Verification]
The \code{CheckNativeSignatures} method represents the optimized code path used strictly for pure Neo workflows where no EVM payload parsing is required.
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysgreen!80!black}]
\item[Usage Scenarios]
Called during the initial Verification phase (transaction validation) filtering to efficiently reject unauthenticated payloads early, and inside the Application execution phase for pure native operations without meta-data overhead.

\item[Mechanism and Limitations]
Leverages \code{Runtime.CheckWitness} directly on the node layer. Since it only targets the signature evidence loaded in the \code{Witnesses} array, it mathematically cannot evaluate EVM typed-data signatures encoded into execution arguments.
\end{description}
\end{featurebox}

\subsection{CheckMixedSignatures: Unified Threshold Algorithm}

The core innovation of the mixed authorization protocol is the \code{CheckMixedSignatures} algorithm, which unifies Neo witness verification and EVM signature verification into a single threshold check. This algorithm is implemented in \code{AbstractAccount.ExecutionAndPermissions.cs:281-315}.

\textbf{Algorithm Definition:}
\begin{codebox}[language=Java]
CheckMixedSignatures(roles, threshold, verifiedSigners):
  Input:  roles          - List of authorized addresses (UInt160[])
          threshold      - Minimum required signatures (int)
          verifiedSigners - EVM addresses recovered from signatures (UInt160[])
  Output: Boolean indicating whether threshold is satisfied
  
  count <- 0
  for each address in roles:
    matched <- false
    
    // Step 1: Check if address is an EVM signer
    if verifiedSigners != null:
      for each signer in verifiedSigners:
        if address == signer:
          count <- count + 1
          matched <- true
          break
          
    // Step 2: Check if address is a Neo signer
    if not matched:
      if Runtime.CheckWitness(address):
        count <- count + 1
        
    // Early exit if threshold reached
    if count >= threshold:
      return true
      
  return false
\end{codebox}

\begin{conceptbox}[Unified Counting Mechanics]
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysblue!80!black}]
\item[Cryptographic Isolation]
EVM signatures are authenticated independently using secp256k1 and Keccak256 logic inside the contract application context. Neo witnesses leverage the node's native secp256r1 core mechanism. The two validation paths cannot intercept, override, or spoof one another.

\item[Equitable Contribution]
Regardless of their origin, each cryptographically validated key mapped to a known account role contributes exactly one unit of authorization weight against the unified threshold.

\item[Order Independence]
The counter algorithm loops over the canonical, on-chain stored array of authorized roles, evaluating valid presence within either the recovered EVM signing set or the Neo environment. This ensures policy behavior is completely agnostic to payload insertion order.
\end{description}
\end{conceptbox}

\subsection{EIP-712 Signature Verification Details}

The EIP-712 verification process ensures that EVM signatures are cryptographically bound to the specific operation and cannot be replayed or misused.

\textbf{Domain Separator Construction:}

\begin{small}
\begin{align*}
\text{domainSep} &= \text{keccak256}(\text{EIP712\_DOMAIN\_TYPEHASH} \\
&\quad \| \text{keccak256}(\text{"Neo N3 Abstract Account"}) \\
&\quad \| \text{keccak256}(\text{"1"}) \\
&\quad \| \text{uint256}(\text{chainId}) \\
&\quad \| \text{address}(\text{verifyingContract}))
\end{align*}
\end{small}

where $\text{chainId} = 894710606$ for Neo N3 testnet and $\text{verifyingContract}$ is the master contract hash.


\subsection{State Management and Nonce Mechanism}

The mixed authorization protocol maintains an \emph{account-scoped} meta-transaction nonce. Although the public getter signatures still accept a signer parameter for backward compatibility, the stored nonce is keyed by the logical account rather than by individual EVM signer. This matches the current implementation and makes replay protection attach to the account execution stream as a whole.

\textbf{Nonce and Activity State.}

For each abstract account, the master contract stores:
\begin{itemize}
\item \textbf{Meta-transaction nonce:} a single monotonically increasing value under the account's nonce storage key.
\item \textbf{Last activity timestamp:} updated whenever the account successfully satisfies an authorization path.
\item \textbf{Temporary execution context:} scoped verify-context and recovered-signer context used only during an active execution path.
\end{itemize}

\textbf{Nonce Validation Flow.}

When processing an EVM meta-transaction, the master contract:
\begin{enumerate}
\item Resolves the logical account from the supplied AA address when the address-based wrapper is used.
\item Recomputes the expected \code{argsHash} and EIP-712 digest for that account and target call.
\item Reads the account-scoped stored nonce and requires the provided nonce to match exactly.
\item Verifies the provided secp256k1 signatures, derives recovered signer addresses, and only then increments the stored account nonce before dispatch.
\end{enumerate}

This account-scoped design intentionally serializes signed meta-transaction execution for a given AA account. Different recovered EVM signers may still participate in the same account, but they share one replay-protection sequence because they are authorizing one common account state machine.

The exact typed-data structure used by the frontend and SDK is:

\begin{codebox}[language=Java]
EIP712Domain {
  name: "Neo N3 Abstract Account"
  version: "1"
  chainId: <Runtime.GetNetwork()>
  verifyingContract: <masterContractHash>
}

MetaTransaction {
  accountAddress: address
  targetContract: address
  methodHash: bytes32
  argsHash: bytes32
  nonce: uint256
  deadline: uint256
}
\end{codebox}

Two details are important. First, the signed identity is the AA \emph{address}, not the raw \code{accountId}. Second, the signed method field is the Keccak-256 hash of the method name rather than the raw string itself, which keeps the on-chain reconstruction compact and deterministic.

The \code{argsHash} is computed by serializing the method arguments with Neo's \code{StdLib.Serialize} and hashing the serialized bytes with Keccak-256, exactly matching the contract's \code{computeArgsHash} helper. After successful verification, the account-scoped nonce is incremented: $n \gets n + 1$.

The \code{deadline} is normalized to Neo's millisecond-based \code{Runtime.Time}. Frontends often sign unix-second deadlines; the contract accepts those inputs by normalizing them before comparison and rejects the meta-transaction whenever:
$$\text{Runtime.Time} > \text{deadline}_{\text{normalized}}$$

\section{Security Analysis}

\subsection{Threat Model}

The Adversary $\mathcal{A}$ is assumed to control the network messaging layer, can observe all mempool transactions, and can submit arbitrary payloads, but computationally cannot:
\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item Break the discrete logarithm assumption.
\item Forge ECDSA signatures (on either secp256k1 or secp256r1 curves) without private keys.
\item Find hash collisions in SHA-256 or Keccak-256.
\end{itemize}

\subsection{Formal Security Definitions}

\begin{definition}[Signature Unforgeability Game]
Adversary $\mathcal{A}$ wins the game if:
\begin{enumerate}[leftmargin=*,topsep=2pt,itemsep=1pt,label=\textbf{\arabic*.}]
\item $\mathcal{A}$ receives target public keys $\{pk_1, \ldots, pk_n\}$.
\item $\mathcal{A}$ makes adaptive signing queries $q_1, \ldots, q_m$.
\item $\mathcal{A}$ outputs a message and signature set $(m^*, \{\sigma_i^*\})$ such that no query was made for $m^*$.
\item Verification accepts $(m^*, \{\sigma_i^*\})$ while having less valid signatures than the threshold $\tau$ (e.g., $|\{\sigma_i^*\}| < \tau$).
\end{enumerate}
The advantage is defined as: $\text{Adv}_{\mathcal{A}} = \Pr[\mathcal{A} \text{ wins}]$.
\end{definition}

We define the security goals of \sys\ through a series of structural assumptions and derived theorems. Table~\ref{tab:security_properties} summarizes the primary security constraints and how they are satisfied by the design.

\begin{theorem}[Threshold Correctness]
Under standard ECDSA collision-resistance assumptions, the master contract's authorization engine accepts an operation if and only if the valid and unique signer count $n \geq \tau$.
\end{theorem}
\begin{proof}[Proof Sketch]
The \code{CheckMixedSignatures} procedure normalizes signers into a single domain by evaluating the set $\mathcal{R}_A \cup \mathcal{R}_M$. For Neo witnesses, \code{CheckWitness} guarantees that the witness contains a valid signature. For EVM signatures, the contract derives $H(m)$ and ecrecover confirms exactly one Ethereum address per signature tuple $(r,s,v)$. Because the mapping step prevents duplicates, the contract accumulates exactly the count of verified members toward the threshold limit.
\end{proof}

\begin{theorem}[Replay Bound]
An intercepted EVM signature $\sigma = (r, s, v)$ constructed for state $St_i$ cannot be re-applied to construct a valid evaluation at state $St_{j}$ where $j > i$.
\end{theorem}
\begin{proof}[Proof Sketch]
The EIP-712 MetaTransaction payload binds $\sigma$ to an account nonce $n_i$. Upon successful verification of the execution block in $St_i$, the contract strictly enforces $n_{i+1} = n_i + 1$. Any subsequent evaluation $St_{j} (j > i)$ will demand $n_j \ge n_{i+1}$. Since $n_i \ne n_j$, the signature verification on $\sigma$ will fail on the fresh typed data hash.
\end{proof}


\section{Related Work}

\subsection{Account Abstraction Systems}

\textbf{ERC-4337.} Ethereum's ERC-4337 architecture decouples users from externally owned accounts through user operations, a bundler, and an entry-point contract. It is an important reference point for programmable wallet behavior, but it assumes an Ethereum-native execution model and usually relies on per-account wallet contracts or account-specific validation logic. By contrast, \sys concentrates authorization in a shared Neo contract and is designed explicitly for heterogeneous Neo/EVM signer sets.

\textbf{Argent and Safe.} Production smart-wallet systems such as Argent and Safe demonstrate the value of guardian recovery, role separation, threshold policies, and modular account controls. Their relevance to \sys is architectural rather than literal: they inform the recovery and governance surface, but they remain single-ecosystem designs and do not solve the problem of counting Neo witnesses and EVM typed signatures inside one threshold rule.

\subsection{Threshold and Cross-Ecosystem Authorization}

\textbf{Threshold-signature literature.} Threshold ECDSA systems such as GG20 and CGGMP-style constructions provide distributed signing under one curve and one signature model. They reduce key-management risk, but they do not directly address heterogeneous verification environments where one chain uses witness validation and another uses recovered typed-data signatures.

\textbf{Bridge and interoperability systems.} Most cross-chain systems focus on message passing or asset transfer. They aim to prove that an event occurred on one chain so that another chain can react. \sys instead targets authorization interoperability: one account policy should accept signers originating from different wallet ecosystems without forcing those ecosystems to share a signature format or a bridging trust assumption.

\subsection{Positioning of Neo-AA}

Relative to the above systems, \sys occupies a distinct design point: zero per-account deployment on Neo N3, deterministic address generation, mixed-signature threshold counting, and a recovery layer that is extensible without changing the primary execution interface. Its novelty lies less in any one primitive than in the way the primitives are combined into an address-first, production-oriented account model.

\section{Discussion}

\subsection{Limitations}

\begin{warningbox}[Current System Limitations]
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysorange!80!black}]
\item[Curve Coverage]
The present implementation supports Neo's native secp256r1 witness path and EVM-style secp256k1 typed-data verification. Supporting additional ecosystems (e.g., Ed25519) would require new signature-verification routines and new serialization or domain-separation rules.

\item[Serialized Meta-Transaction Sequencing]
The current account-scoped nonce is simple and safe, but it also means that meta-transactions for one account advance through one shared replay-protection sequence. This is a conscious safety-oriented choice rather than an oversight.

\item[Execution Cost]
EVM public-key recovery and signature verification add computational overhead relative to pure Neo witness validation. The design remains practical for the intended governance and account-operation workflows, but it is not entirely free.
\end{description}
\end{warningbox}

\subsection{Design Trade-offs}

\begin{conceptbox}[Architectural Trade-offs]
\begin{description}[leftmargin=*,style=nextline,font=\bfseries\color{sysblue!80!black}]
\item[Shared Contract vs. Per-Account Deployment]
A shared contract minimizes deployment cost and simplifies upgrades to common logic, but pushes more complexity into deterministic routing, scoped execution contexts, and careful internal state separation.

\item[Address-First Usability vs. Logical-ID Storage]
Exposing the AA address as the primary public identity improves user experience and frontend ergonomics, while retaining \code{accountId} internally keeps storage, indexing, and verifier integration straightforward. The specification treats identifiers as complementary.

\item[Unified Verification vs. Separate Execution Planes]
Counting Neo witnesses and EVM recovered signers in one threshold path yields a simple and expressive policy model, but requires a two-phase verification loop and careful handling of temporary script contexts.
\end{description}
\end{conceptbox}

\subsection{Operational Architecture}

\begin{featurebox}[Modular Contract Structure]
The deployed master contract follows a deliberately modular split to maintain maintainability despite its shared nature:
\begin{itemize}[leftmargin=*,topsep=2pt,itemsep=1pt]
\item \textbf{\texttt{AccountLifecycle}}: Account creation, validation, and deterministic address binding.
\item \textbf{\texttt{StorageAndContext}}: Canonical storage keys, address mappings, activity tracking, verify context, and execution locks.

\subsection{Future Directions}

Promising extensions include broader ecosystem support (for example, additional signature curves and domain formats), privacy-preserving authorization with zero-knowledge techniques, richer batch execution patterns, and machine-checked verification of the full execution-context invariants. Another natural direction is stronger domain-centric discovery so that \code{.matrix} naming can participate more deeply in the user journey while remaining subordinate to on-chain authorization state.

\subsection{Deployment Details}

The production-oriented implementation is validated against Neo N3 testnet workflows and is accompanied by reproducible contract artifacts, integration scripts, and frontend support for collaborative transaction preparation. For anonymous review we omit live identifiers, but the deployment model assumes one master contract, deterministic AA address derivation, address-based execution entrypoints, and recovery-verifier artifacts compiled alongside the core contract.



\section{Conclusion}

We presented \sys, a Neo N3 account-abstraction architecture that unifies native witness-based authorization with EVM EIP-712 typed signatures inside one shared master contract. The core contribution is not merely signature verification, but a complete account model that combines deterministic AA addressing, address-first user interaction, mixed-threshold authorization, and extensible recovery under a single on-chain control plane.

The polished specification now makes two implementation-aligned points explicit. First, the externally meaningful identity is the deterministic AA address, while \code{accountId} remains an internal key used for storage, deterministic routing, and verifier compatibility. Second, the mixed-authorization path is account-centric: recovered EVM signers and Neo witnesses are normalized into one permission engine, and replay protection advances through an account-scoped nonce rather than signer-specific counters.

Taken together, these choices make \sys practical as well as conceptually clean. One master contract can support many accounts without per-account deployment, users can discover and operate accounts through AA addresses or \code{.matrix} names, and recovery logic can evolve without redesigning the execution surface. Future work includes broader ecosystem support, privacy-preserving authorization, and more formal verification of the execution-context invariants that underpin the system's safety.
